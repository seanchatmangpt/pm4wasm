# Vision 2030: POWL Rust/WASM and the Future of Browser-Native Process Mining

## Where We Are Today

The POWL Rust/WASM crate delivers a complete implementation of the POWL v2 language compiled to WebAssembly:

- **Parsing** — POWL model strings parsed entirely in the browser
- **Validation** — Strict partial order constraints verified at O(n³) with bit-packed matrices
- **Conversion** — Models converted to Petri nets and process trees for downstream analysis
- **Footprints** — Behavioral signatures extracted without a Python server
- **Simplification** — Structural normalization applied client-side

This is a foundation. The following describes where it can go.

## 2025.5 — NL-to-POWL-to-BPMN (COMPLETE)

### AI-Assisted Process Discovery from Natural Language

The central paradigm shift: process models generated from natural language descriptions, verified for structural soundness, and exported to BPMN 2.0.

```python
from pm4py.algo.dspy.powl.natural_language import generate_powl_from_text
from pm4py.objects.powl.parser import parse_powl_model_string

result = generate_powl_from_text(
    "A customer submits an order. The order is validated. "
    "If valid, it is picked, packed, and billed in parallel. "
    "If invalid, it is cancelled and refunded."
)
# Returns: {powl: str, verdict: bool, reasoning: str, refinements: int}
```

**Key components:**
- **NaturalLanguageToPOWL** — DSPy ReAct agent that generates POWL from NL
- **POWLJudge** — "Dr. van der Aalst" evaluates structural soundness (no ground truth)
- **Judge-refinement loop** — rejected models get feedback and re-generate
- **Few-shot demos** — 4 domain examples teaching correct XOR vs PO operator selection
- **BPMN round-trip** — verified POWL → BPMN 2.0 XML (Camunda/Signavio compatible)

**CLI:**
```bash
python -m pm4py.cli DiscoverPOWLToBPMN "process description..." output.bpmn
```

## 2025 — Completeness

### Event Log Processing in WASM
Bring the pm4py event log reader (CSV/XES import) into a second WASM module `pm4py-log-wasm`. A user drops a 50 MB event log onto a web page and it is parsed and filtered in the browser — no upload, no server, no privacy risk.

**Key deliverable:** `parse_xes()`, `parse_csv()`, `filter_variants()` exported as WASM functions.

### Conformance Checking
Port the token-based replay and alignments algorithms to Rust. Given a `PetriNetResult` and an event log, compute conformance fitness without leaving the browser.

```javascript
const fitness = compute_fitness(petri_net_result, event_log);
console.log(fitness.percentage);  // 0.94
```

### npm Package
Publish `pm4wasm` to npm with TypeScript types generated by `wasm-bindgen`. Install with:


## 2026 — Discovery

### POWL Discovery from Event Logs in WASM
Port the SplitMiner / IMf discovery algorithms to Rust so that a POWL model can be discovered directly from an event log inside the browser.

```javascript
const log = await parse_xes(file);
const model = discover_powl(log, { noise_threshold: 0.2 });
console.log(powl_to_string(model));
```

**Impact:** A practitioner with a laptop and no Python installation can go from raw XES to a discovered POWL model in one drag-and-drop.

### Incremental Discovery
Stream an event log chunk by chunk — update the discovered model incrementally. Enables real-time process mining dashboards that update as new events arrive.

```javascript
const discoverer = new IncrementalPowlDiscoverer({ noise_threshold: 0.2 });

for await (const batch of event_stream) {
  discoverer.update(batch);
  renderModel(discoverer.get_current_model());
}
```

## 2027 — Scale

### SIMD Acceleration
The bit-packed `BinaryRelation` already positions us for SIMD. When browsers universally support WebAssembly SIMD (already shipping in Chrome and Firefox), recompile with `--target-features=+simd128`. The inner Warshall loop becomes 4× faster by operating on 128-bit vectors instead of 64-bit words.

```
Before SIMD: n=200 closure in 8 ms
After SIMD:  n=200 closure in 2 ms  (estimated)
```

### Multi-threaded WASM
When `SharedArrayBuffer` restrictions are relaxed and `wasm-threads` stabilizes:
- Parallelize transitive closure across rows (embarrassingly parallel outer loop)
- Run conformance checking for multiple traces simultaneously
- Target: 16-core browser worker pool reducing O(n³) wall time by 16×

### Server-Side Rust via Native Binaries
The `rlib` target already supports native compilation. Publish a PyO3-based `pm4py-rust` Python extension that replaces the pure-Python `BinaryRelation.py` with zero API changes:

```python
# Drop-in replacement — same API, 50x faster
from pm4py_rust import BinaryRelation  # backed by Rust
```

## 2029 — Ecosystem

### Federated Process Mining
An organization with strict data residency requirements runs process mining entirely in-browser or on-device. The WASM module is the trusted compute boundary:

- Event log stays on the client
- Discovered model is the only artifact that leaves the device
- Conformance statistics are computed locally and only aggregated summaries are shared

This enables cross-organization process benchmarking without sharing sensitive event data.

### POWL as an Interchange Standard
Promote the POWL string format (currently the Python `__repr__`) as a lightweight, human-readable interchange format for process models alongside BPMN XML and PNML:

```
PO=(nodes={ A, X(B, C), D }, order={ A-->X(B,C), X(B,C)-->D })
```

One line, version-controllable in git, parseable in Rust and Python with the same grammar.

### Plugin Architecture
Allow third-party algorithms to be packaged as WASM modules that plug into the core runtime:

```javascript
import { load_plugin } from '@pm4py/pm4wasm';
import myConformancePlugin from './my-plugin.wasm';

await load_plugin(myConformancePlugin);

// Plugin's exported function now available
const result = my_conformance_check(model, log);
```

## Technical Principles for the Journey

| Principle | Implication |
|-----------|-------------|
| **Single source of truth** | One Rust codebase, compiled to WASM and PyO3; no duplicated logic |
| **API stability** | `parse_powl()` / `powl_to_string()` round-trip must never break |
| **Zero copies where possible** | WASM linear memory → JS ArrayBuffer without intermediate allocations |
| **Test parity** | Every algorithm tested in native Rust and WASM; no browser-only surprises |
| **Progressive enhancement** | Works in any browser that supports WASM; SIMD and threads are opt-in enhancements |
